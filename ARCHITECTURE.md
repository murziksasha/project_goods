# Architecture

## Обзор

Project Goods организован как full-stack приложение из трех основных частей:

- React frontend для рабочего интерфейса
- Express backend с REST API под префиксом `/api`
- MongoDB как основное хранилище данных

## Общий поток данных

```text
Браузер
   |
   v
Frontend (React + Axios)
   |
   v
Backend API (Express routes)
   |
   v
Domain services
   |
   v
Mongoose models
   |
   v
MongoDB
```

## Архитектура frontend

Frontend использует слоистую структуру, близкую к feature-sliced подходу:

- `app` инициализирует приложение
- `pages` определяют страницы верхнего уровня
- `widgets` собирают крупные бизнес-секции интерфейса
- `features` описывают пользовательские действия
- `entities` инкапсулируют доменные типы, API и UI-фрагменты
- `shared` содержит общие утилиты, стили и инфраструктуру

### Доступ к данным на frontend

- HTTP-запросы идут через Axios
- `VITE_API_URL` определяет base URL backend
- доменные API-модули лежат рядом с соответствующими сущностями

### Оркестрация состояния

Главной точкой композиции интерфейса сейчас выступает dashboard-страница. Она координирует:

- товары и продажи
- заказы на услуги
- сотрудников
- настройки
- финансовые действия
- уведомления и пользовательскую обратную связь

## Архитектура backend

Backend собран по модульному практическому паттерну:

- `app.ts` настраивает Express, middleware и подключение роутеров
- `routes/` описывает HTTP-контракты
- `domain/*/service.ts` содержит прикладную логику и use cases
- `domain/*/model.ts` определяет модели хранения
- `config/` отвечает за окружение и подключение к БД
- `shared/` хранит общие утилиты и тестовые данные

### Характеристики API

- REST API на JSON
- CORS управляется через `CLIENT_ORIGIN`
- ошибки централизованно обрабатываются middleware-слоем
- есть health endpoint для быстрой проверки окружения

## Доменные области

Текущая архитектура строится вокруг следующих областей:

- products
- clients
- sales
- employees
- settings
- finance
- demo data

## Хранение данных

- основное хранилище - MongoDB
- слой доступа построен на Mongoose
- для последовательностей и служебной генерации значений используется отдельный домен `sequence`

## Локальная инфраструктура

`docker-compose.yml` поднимает MongoDB для локальной разработки.

Это дает простой локальный стек:

- не нужен внешний облачный сервис для старта
- backend подключается к базе через `MONGO_URI`

## Обработка ошибок

- validation errors и duplicate key ошибки возвращаются как `400`
- неожиданные ошибки возвращаются как `500`
- сервер логирует неожиданные backend-ошибки в консоль

## Сильные стороны текущей архитектуры

- хорошо разделены UI, transport layer и бизнес-логика
- backend уже разложен по доменам
- frontend структура подходит для постепенного роста
- локальный запуск остается легким

## Текущие архитектурные пробелы

- нет автоматических тестов
- нет общей схемы валидации между frontend и backend
- нет формального workspace-инструмента для монорепо
- импорт товаров из Excel пока только зарезервирован интерфейсно

## Рекомендуемое развитие

1. Добавить автоматические тесты для доменных сервисов и критичных UI-сценариев.
2. Усилить валидацию входных данных на границах API.
3. Зафиксировать общие стандарты linting, formatting и quality gates.
4. Рассмотреть workspace-tooling, если общих скриптов и связей между пакетами станет больше.
